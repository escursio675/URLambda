# Database Schema Documentation

## 1. Overview

The application uses **MongoDB** as the primary NoSQL database and **Mongoose ODM** for schema modeling and database interaction.

The database currently consists of two primary collections:

1. `urls`
2. `users`

The schema design follows a document-oriented architecture optimized for:

- Fast URL lookup
- Efficient redirection
- User-based URL ownership
- Future analytics integration
- Scalable read-heavy workloads

# 2. Entity Relationship Overview

```
User (1) ──────────── (Many) URL
```

A single authenticated user can own multiple shortened URLs.

# 3. URL Collection Schema

## Collection Name

```
urls
```

## Schema Definition

| Field | Type | Constraints | Description |
| --- | --- | --- | --- |
| `_id` | ObjectId | Auto-generated | Unique MongoDB document identifier |
| `shortCode` | String | Required, Unique, Indexed | Unique alphanumeric code used for redirection |
| `shortUrl` | String | Required | Complete shortened URL returned to user |
| `longUrl` | String | Required | Original destination URL |
| `user` | ObjectId | Optional, Reference → User | Owner of the shortened URL |
| `createdAt` | Date | Auto-generated | Timestamp of creation |
| `updatedAt` | Date | Auto-generated | Timestamp of latest update |
| `__v` | Number | Auto-generated | Internal Mongoose version key |

## Schema Implementation

```
consturlSchema=newmongoose.Schema(
{
    shortCode: {
        type:String,
        required:true,
        unique:true,
        index:true
    },

    shortUrl: {
        type:String,
        required:true,
    },

    longUrl: {
        type:String,
        required:true,
    },

    user: {
        type:mongoose.Schema.Types.ObjectId,
        ref:'User',
        default:null
    }
},
{ timestamps:true }
);
```

## Field Explanations

### `shortCode`

A unique 6-character alphanumeric identifier generated by the backend service.

Example:

```
abc123
```

Purpose:

- Used in redirection routing
- Indexed for fast database retrieval
- Serves as the lookup key for URL resolution

### `shortUrl`

The complete shortened URL exposed to the user.

Example:

```
https://urlambda.vercel.app/abc123
```

Purpose:

- User-facing shareable URL
- QR code generation input
- Dashboard display

### `longUrl`

Stores the original destination URL.

Example:

```
https://google.com
```

Purpose:

- Redirect destination
- Core persistent mapping target

### `user`

Optional reference to the owner of the shortened URL.

Relationship:

```
Many URLs → One User
```

Purpose:

- Enables dashboard ownership
- Supports authenticated analytics
- Allows future per-user history tracking

## Indexing Strategy

### Indexed Field

```
shortCode
```

Reason:

- Redirection queries are extremely frequent
- Optimizes:

```
Url.findOne({ shortCode })
```

Expected benefit:

- Faster lookup performance
- Reduced query latency
- Improved scalability under high traffic

# 4. User Collection Schema

## Collection Name

```
users
```

## Schema Definition

| Field | Type | Constraints | Description |
| --- | --- | --- | --- |
| `_id` | ObjectId | Auto-generated | Unique MongoDB document identifier |
| `googleId` | String | Required, Unique, Indexed | Google OAuth identifier |
| `email` | String | Required, Unique | User email address |
| `name` | String | Required | User display name |
| `avatar` | String | Optional | Profile image URL |
| `createdAt` | Date | Auto-generated | Account creation timestamp |
| `updatedAt` | Date | Auto-generated | Latest update timestamp |
| `__v` | Number | Auto-generated | Internal Mongoose version key |

## Schema Implementation

```
constuserSchema=newmongoose.Schema(
{
    googleId: {
        type:String,
        required:true,
        unique:true,
        index:true
    },

    email: {
        type:String,
        required:true,
        unique:true,
        lowercase:true
    },

    name: {
        type:String,
        required:true
    },

    avatar: {
        type:String
    }
},
{ timestamps:true }
);
```

## Field Explanations

### `googleId`

Unique identifier returned by Google OAuth authentication.

Purpose:

- User authentication
- Session association
- Prevent duplicate accounts

### `email`

Stores the authenticated user's email address.

Properties:

- Automatically converted to lowercase
- Enforced uniqueness

Purpose:

- Account identification
- Future notifications/contact

### `name`

Stores the user’s display name retrieved during authentication.

Purpose:

- Dashboard personalization
- User identification

### `avatar`

Stores the URL of the user's profile image.

Purpose:

- Dashboard UI personalization
- Account representation

## Indexing Strategy

### Indexed Field

```
googleId
```

Reason:

- Frequently queried during OAuth login flow
- Enables rapid authentication lookup

# 5. Relationship Design

## One-to-Many Relationship

```
User → URLs
```

Implementation:

```
user: {
type:mongoose.Schema.Types.ObjectId,
ref:'User'
}
```

Meaning:

- One user may own multiple URLs
- Each URL belongs to at most one user

# 6. Database Design Rationale

The schema design prioritizes:

- Fast redirection performance
- Minimal query complexity
- Scalability for read-heavy workloads
- Simplicity of document retrieval
- Modular extensibility for analytics and dashboards

MongoDB was selected because:

- URL documents are naturally document-oriented
- Schema evolution is flexible
- Read performance is efficient
- Integration with Node.js and Mongoose is straightforward
